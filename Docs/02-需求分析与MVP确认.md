# 需求分析与 MVP 确认 · 图书馆座位预约系统

---

## 1. 目标用户与核心场景

### 1.1 用户角色定义

| 角色 | 真实身份 | 使用频率 | 技术能力 | 使用设备 |
|------|----------|----------|----------|----------|
| **学生** | 在校本科生/研究生 | 每天 1-3 次（考试季更频繁） | 会用浏览器，不要求技术背景 | 个人笔记本 + 手机浏览器 |
| **管理员** | 图书馆工作人员 / 勤工助学学生 | 每天不定时查看 | 基础电脑操作 | 图书馆前台电脑 |

### 1.2 核心场景描述

**场景一：考试季学生提前预约座位**

> 周五晚上，学生小李打算周六去图书馆复习。他打开座位预约系统，看到三楼阅览区还有 12 个空闲座位。他选了一个靠窗的座位 A-12，预约时段为 9:00-12:00。提交后系统提示预约成功。第二天他到馆直接落座。

**场景二：学生临时到馆找座**

> 学生小王下课后来到图书馆，在一楼大厅打开系统查看空闲座位。发现二楼还有 3 个空位，他预约了其中一个，看到座位号后直接上楼找到座位。

**场景三：管理员整理座位信息**

> 图书馆管理员张老师发现四楼新增了 10 张桌子共 20 个座位。他登录管理后台，在座位管理中逐个添加新座位，并标注了区域为"四楼自习区"。

**场景四：管理员查看预约数据**

> 学期末，张老师想了解这学期座位使用情况。他登录后台查看统计数据，发现本学期日均预约 80 人次，座位利用率约 65%，三楼阅览区最受欢迎。

## 2. 用户最关心的问题

### 2.1 学生最关心的 5 个问题

| 排名 | 问题 | 对应功能 | 优先级影响 |
|------|------|----------|------------|
| 1 | **现在还有没有空位？** | 座位列表 + 空闲计数 | 核心功能，必须首页可见 |
| 2 | **我选的这个座位到了还能坐吗？** | 预约提交 + 冲突检测 | 预约必须可靠，不能出现多人约同一座位同一时段 |
| 3 | **约错了能不能取消？** | 取消预约 | 必须支持，否则一个错误预约就浪费一个座位 |
| 4 | **我约了哪些座位？什么时候的？** | 我的预约 | 必须让学生能看到自己的所有预约 |
| 5 | **哪个区域比较安静/有空位？** | 区域筛选 | 提升体验，但不是必须 |

### 2.2 管理员最关心的 3 个问题

| 排名 | 问题 | 对应功能 | 优先级影响 |
|------|------|----------|------------|
| 1 | **今天有多少人预约了？哪些座位被占了？** | 预约管理 + 统计 | 核心管理需求 |
| 2 | **有人预约了不来怎么办？** | 手动取消预约 | 至少支持管理员干预 |
| 3 | **新加的座位怎么录进系统？** | 座位管理 CRUD | 基础维护功能 |

## 3. 候选功能池

基于校园图书馆场景和同类预约系统的常见能力，梳理出以下候选功能。所有功能分为"用户端"和"管理端"两组。

### 3.1 用户端候选功能

| 编号 | 功能 | 简要说明 |
|------|------|----------|
| UF-01 | 体验账号模拟登录 | 首页提供多个体验学生入口（学生A/B/C），点击即切换身份，无需密码 |
| UF-02 | 空闲座位概览 | 首页顶部显示当前总座位数、空闲座位数 |
| UF-03 | 座位列表 | 以卡片或表格展示所有座位，显示编号、区域、状态 |
| UF-04 | 区域筛选 | 按区域（一楼大厅、二楼阅览区、三楼自习区等）过滤座位列表 |
| UF-05 | 座位详情 | 点击单个座位查看详细信息：楼层、区域、座位号、设施描述 |
| UF-06 | 预约提交 | 选择座位 + 日期 + 时段 + 提交预约 |
| UF-07 | 预约冲突检测 | 提交时检查同一座位同一时段是否已被他人预约 |
| UF-08 | 我的预约 | 当前学生的所有预约记录列表 |
| UF-09 | 取消预约 | 在自己的预约列表中取消某条"预约中"的记录 |
| UF-10 | 预约时长限制 | 单次预约最长 4 小时，每天最多预约 2 次 |
| UF-11 | 座位状态颜色标识 | 列表页中空闲/预约中/使用中以不同颜色区分 |

### 3.2 管理端候选功能

| 编号 | 功能 | 简要说明 |
|------|------|----------|
| AF-01 | 管理员登录 | Session/Cookie 登录，预置 admin/123456 |
| AF-02 | 全部预约列表 | 查看所有学生的预约，按状态/日期筛选 |
| AF-03 | 管理员取消预约 | 管理员手动取消任意预约 |
| AF-04 | 座位新增 | 添加新座位（编号、区域、楼层、描述） |
| AF-05 | 座位编辑 | 修改已有座位信息 |
| AF-06 | 座位删除 | 删除座位（有未完成预约的座位不可删除） |
| AF-07 | 统计页 | 总座位数、今日预约数、座位利用率、热门区域排名 |
| AF-08 | 数据导出 | 将预约记录导出为 CSV 或 Excel |

## 4. 功能优先级矩阵

### 4.1 优先级分层标准

| 层级 | 名称 | 含义 | 可接受延迟 |
|------|------|------|------------|
| **P0** | 核心基础 | 缺了系统跑不起来，或者核心流程断裂 | 不可延迟 |
| **P1** | 主体业务 | 核心流程能走通但功能残缺，体验断档 | 最多延迟到 M3 |
| **P2** | 体验提升 | 有更好，没有不影响业务流程 | 可延迟到 V1.1 |
| **P3** | 暂缓 | 本期不做 | 至少 V1.2 以后 |

### 4.2 完整优先级矩阵

| 编号 | 功能 | 层级 | 工作量 | 理由 |
|------|------|------|--------|------|
| UF-01 | 体验账号模拟登录 | **P0** | 小 | 没有登录入口，学生无法使用系统任何功能 |
| UF-03 | 座位列表展示 | **P0** | 中 | 核心信息展示，没有座位列表就没有预约对象 |
| UF-06 | 预约提交 | **P0** | 大 | 核心业务动作，系统的价值出口 |
| UF-07 | 预约冲突检测 | **P0** | 中 | 没有冲突检测，两个学生可能约到同一座位同一时段 |
| UF-08 | 我的预约 | **P0** | 中 | 学生必须能看到自己约了什么，是信任基础 |
| UF-09 | 取消预约 | **P0** | 中 | 不能取消的预约是陷阱，会浪费座位 |
| AF-01 | 管理员登录 | **P0** | 小 | 管理员入口 |
| AF-02 | 全部预约列表 | **P0** | 中 | 管理员必须能看到所有预约才能管理 |
| AF-03 | 管理员取消预约 | **P0** | 小 | 处理违规或异常预约的必要手段 |

| AF-07 | 统计页 | **P1** | 中 | 有价值但非核心管理动作 |
| AF-05 | 座位编辑 | **P2** | 小 | 初始数据正确后编辑频率很低 |

## 5. MVP 范围确认

### 5.1 MVP 包含范围

MVP = **P0 全部 9 项 + P1 中 4 项（UF-02、UF-04、UF-05、UF-11）**

共 **13 项功能**，按工程合并后实际对应 8 个页面 + 1 个共享布局：

| 页面 | 对应功能 | 角色 |
|------|----------|------|
| 用户首页（含空闲概览 + 体验账号入口） | UF-01, UF-02 | 学生 |
| 座位列表（含区域筛选 + 状态颜色） | UF-03, UF-04, UF-11 | 学生 |
| 座位详情（独立页面或弹窗） | UF-05 | 学生 |
| 预约提交页（含冲突检测） | UF-06, UF-07 | 学生 |
| 我的预约（含取消操作） | UF-08, UF-09 | 学生 |
| 管理员登录页 | AF-01 | 管理员 |
| 预约管理页（含查看/筛选/取消） | AF-02, AF-03 | 管理员 |
| 统计页 | AF-07 | 管理员 |

### 5.2 MVP 不包含（明确暂缓）

以下功能属于"可以做但本期不做"，在 PRD Lite 中不需要写页面设计：

| 功能 | 暂缓原因 | 计划版本 |
|------|----------|----------|
| AF-05 座位编辑 | 初始种子数据准确的话，MVP 阶段几乎不需要编辑 | V1.1 |
| AF-06 座位删除 | 可先通过修改状态标记停用 | V1.1 |
| UF-10 预约时长限制 | 课堂演示场景下用户可以自觉遵守 | V1.1 |
| AF-04 座位新增（从 MVP 移到暂缓） | 种子数据初始化 30+ 座位，MVP 演示足够 | V1.1（如需） |
| AF-08 数据导出 | 课堂提交只需演示 + 录屏，不需要导出文件 | V1.2+ |

### 5.3 MVP 可行性论证

| 维度 | 评估 |
|------|------|
| **页面数量** | 6（用户端）+ 2（管理端）+ 1（Layout）= **9 个视图文件** |
| **数据表数量** | 3 张表：Seats、Reservations、Users（或直接用 Session 模拟） |
| **核心复杂度** | 预约冲突检测是最复杂的一段逻辑，其余均为标准 CRUD |
| **单人 4 周评估** | 第 1 周搭架子（数据模型 + 种子 + 2 页），第 2 周完成用户端（4 页），第 3 周管理端 + 冲突检测优化，第 4 周收尾 |
| **结论** | **可行，但第 2 周不能拖延。** 如果第 2 周结束时我的预约 + 取消预约未完成，第 3 周管理端需要削减页面 |

## 6. 本期明确不做的功能

以下功能来自立项单第 6 节"暂不开发内容"，经需求分析确认后维持不变：

### 绝对不做（无论时间是否充裕）

- ❌ **支付功能**：本项目不涉及任何金额
- ❌ **扫码签到/签退**：需要硬件支持，课堂无法演示
- ❌ **消息推送**：需要短信/邮件网关或 WebSocket，复杂度超出课程范围
- ❌ **复杂账号体系**：注册、找回密码、手机绑定、第三方登录全部排除
- ❌ **多馆区联动**：本期只做一个馆的单层座位

### 可能想做但确认不做（经过分析后排除）

- ❌ **实时座位图（Canvas/SVG 平面图）**：虽然效果好，但开发量远超表格展示，且单人 4 周工期不支持
- ❌ **违规计分/黑名单机制**：需要长期运行积累数据才有意义，演示时看不到效果
- ❌ **小程序 / 移动 App**：本课程只要求 ASP.NET Core MVC，不涉及移动端开发
- ❌ **多语言 / 主题切换**：与核心业务无关

## 7. 需求评审结论

### 7.1 评审信息

| 项目 | 内容 |
|------|------|
| 评审时间 | 2026-07-07 |
| 参与角色 | 开发（实习生 1 人）、项目经理助理（模拟）、产品助理（模拟） |
| 评审方式 | 桌面推演 |

### 7.2 评审讨论要点

**争议点 1：座位详情是否需要独立页面？**

- 产品助理意见：详情页可以展示座位照片、设施描述、预约按钮，独立页面更清晰
- 开发意见：如果独立页面，需要多一个路由 + 一个视图文件；如果做成弹窗，可以在座位列表页完成
- **决定**：座位详情做成 Bootstrap 弹窗（Modal），不新增独立页面。节省的时间用于做好冲突检测

**争议点 2：统计页做到什么程度？**

- 产品助理意见：最好有柱状图/饼图展示热门区域排名
- 开发意见：图表需要引入 Chart.js 或其他库，增加前端复杂度
- **决定**：MVP 不做图表，只做纯数字卡片（总座位数、今日预约数、座位利用率、热门区域文字列表）。图表作为 P2 体验提升项

**争议点 3：区域筛选用什么 UI？**

- 决定：下拉选择框（Dropdown），最简单、最稳定、不需要额外 JS 库

### 7.3 保留项、删减项、暂缓项

| 类别 | 功能 | 处理方式 |
|------|------|----------|
| **保留项（MVP 确认）** | UF-01~UF-09（学生端）+ AF-01~AF-03（管理端登录+预约管理）+ UF-11（状态颜色标识）+ AF-07（纯数字统计页） | 全部进入 PRD Lite |
| **删减项（从立项单中移除）** | AF-04 座位新增（改为种子数据预置，管理端暂不做 CRUD） | 不在 PRD Lite 中出现 |
| **删减项（合并页面）** | UF-05 座位详情（从独立页改为弹窗） | 减少 1 个页面 |
| **暂缓项（V1.1 再议）** | AF-05 座位编辑、AF-06 座位删除、UF-10 预约时长限制 | PRD Lite 中不涉及 |
| **暂缓项（V1.2+）** | AF-08 数据导出 | 不列入本期计划 |

### 7.4 评审结论

**通过。** 以上 MVP 范围（P0 全部 + P1 精选）确认进入 PRD Lite 阶段。总页面数从立项单的 10 页缩减为 8 页，功能聚焦于"学生选座预约"和"管理员管理预约"两条主线。

## 8. 进入 PRD Lite 前的最终口径

### 8.1 一句话定义本期做什么

> 开发一个图书馆座位预约 Web 系统，让学生能在线查看空闲座位、预约座位、取消预约，让管理员能管理预约、查看基本统计。

### 8.2 范围红线

```
【本期必须做】                         【本期绝对不做】
  学生：选座 → 预约 → 查看 → 取消        支付 / 扫码 / 消息推送
  管理员：登录 → 管理预约 → 看统计        注册 / 多馆区 / 实时座位图
  基础：3 张表 + 种子数据 + 模拟登录      图表 / 导出 / 违规计分
```

### 8.3 给 PRD Lite 的输入

以下 5 个决策直接输入 PRD Lite，不需要重新讨论：

| 决策 | 结论 |
|------|------|
| 预约时段粒度 | 按小时段预约（如 09:00-10:00），由用户在预约时选择起止时间 |
| 座位详情 UI | 不独立页面，用 Bootstrap Modal 弹窗承载 |
| 统计页 | 纯数字卡片，不带图表 |
| 区域筛选 | 下拉选择框 |
| 体验学生数量 | 预置 3 个体验账号（学生 A/B/C），仅用于区分不同人的预约数据 |
