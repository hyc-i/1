#### [P0-01] IX_Reservation_Seat_Time 索引唯一性冲突（严重 — 阻塞编码）

**核心冲突**：`IX_Reservation_Seat_Time` 在 4 份文档中的声明相互矛盾。

| 文档 | 索引类型声明 | 具体证据 |
|------|------------|----------|
| **D03-11.3** | **唯一（UNIQUE）** | "(SeatId, ReservationDate, StartTime, EndTime) **唯一组合索引**防止重复预约" |
| **D07-附录A** | **非唯一（默认）** | `HasIndex(...).HasDatabaseName("IX_Reservation_Seat_Time")` — **无 `.IsUnique()`** 调用 |
| **D08-4.2** | **非唯一** | 索引类型列标注为"**非唯一**"；D08-7.3 详细解释了"不能做唯一"的理由 |
| **D08-11.1#4** | **带修正声明** | "PRD 03 第 11.3 节称'唯一组合索引防止重复预约'，此说法有误导" |
| **D09-8.3** | **唯一（UNIQUE）** | "**唯一索引** IX_Reservation_Seat_Time 作为第二道防线" + `catch (DbUpdateException)` |
| **D09-8.3 代码** | **隐含唯一** | `catch (DbUpdateException)` 只有索引为 UNIQUE 时才会触发 |

**矛盾根源分析**：

D08-7.3 正确指出：单独对 (SeatId, ReservationDate, StartTime, EndTime) 建唯一索引不能防御时段重叠（如 09:00~12:00 vs 10:00~11:00），因为四条字段值都不相同。

但 D09-8.3 也正确指出：唯一索引可以防御"完全相同的 (SeatId, ReservationDate, StartTime, EndTime) 组合"在高并发下被插入两次。

两者的论点不矛盾，但**结论矛盾**：D08 说"非唯一"，D09 说"唯一"。

**对编码的实际影响**：

| 场景 | 如果索引为 UNIQUE | 如果索引为非唯一 |
|------|------------------|-----------------|
| EF Core 配置 | 添加 `.IsUnique()` | 不添加（保持默认） |
| 生成的迁移 SQL | `CREATE UNIQUE INDEX` | `CREATE INDEX` |
| DbUpdateException（D09 8.3） | 可以捕获并发重复 | 永远不会触发 |
| 正常冲突检测 | 业务层完成（AnyAsync） | 业务层完成（AnyAsync） |
| 极端并发下一模一样的两条 | 第二条被拒 | 会写入两条完全相同的记录 |

**无论选哪一个，编码前必须统一。** 目前 D07 的 EF Core 代码是最终产出，但 D09 的异常处理代码依赖唯一索引。

**推荐方案**：添加 `.IsUnique()`（采纳 D09 的双层防护策略），同时在 D08-4.2 中增加说明：
> "虽然唯一索引不能防御时段重叠，但可以防止完全相同的参数在高并发下被重复写入。索引角色修改为唯一索引，冲突检测的业务逻辑不变。"

**修改量**：
- D07-附录A 第 544 行：`HasIndex(...)` 后加 `.IsUnique()`
- D08-4.2 索引类型：`非唯一` → `唯一`，同时在 7.3 节补充"唯一索引仍然有助于防御完全重复参数"的说明
- D09-8.3：不需改（已正确描述唯一索引策略）

#### [P1-06] D08-5.3 StartTime/EndTime 存储规则与 D09 L1-5 时间验证的潜在矛盾

## 9. P0 阻塞项（不解决不能开工）

### P0-01：IX_Reservation_Seat_Time 索引唯一性

**严重程度**：**阻塞** — 直接决定 EF Core 迁移 SQL 的正确性

**各文档立场**：

```
D03-11.3:  "唯一组合索引"  ──→  ✅ UNIQUE
D07-附录A:  无 .IsUnique()  ──→  ❌ 非唯一（默认）
D08-4.2:   "非唯一"        ──→  ❌ 非唯一（明确声明）
D08-7.3:   解释不能唯一的理由 ──→  ❌
D08-11.1#4:指出 D03 有误导   ──→  ⚠️ 带修正声明
D09-8.3:   唯一索引作为第二防线 ──→  ✅ UNIQUE（含 catch DbUpdateException）
```

**必须编码前统一**。否则 EF Core 迁移和并发防护代码无法确定。

**推荐修复方案**（选定为 UNIQUE）：

```
修改 D07-附录A:
  HasIndex(r => new { r.SeatId, r.ReservationDate, r.StartTime, r.EndTime })
    .HasDatabaseName("IX_Reservation_Seat_Time")
    .IsUnique();  ← 添加此行

修改 D08-4.2 索引表:
  IX_Reservation_Seat_Time 的类型从 "非唯一" → "唯一"

修改 D08-7.3:
  增加说明："注：索引同时添加了唯一约束，
  尽管不能防御时段重叠（见上例），但可以防止
  完全相同参数的重复插入，作为第二道并发防线。"

D09-8.3 不需修改（已正确描述唯一索引策略）。
```

**预计修复用时**：15 分钟
