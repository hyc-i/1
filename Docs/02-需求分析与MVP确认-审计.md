# 需求分析与 MVP 确认 · 审计报告

审计对象：Docs/02-需求分析与MVP确认.md  
审计日期：2026-07-07  
审计结论：**有条件通过**，存在 2 项必改 + 3 项建议，修改后方可进入 PRD Lite。

---

## A. 总体判断

这份文档**比立项单有进步，但进步不够大**。它确实新增了一些立项单中没有的内容（用户故事、功能清单、优先级矩阵、MVP 范围定义），形式上完成了"从立项到需求分析"的推进。但仔细读下去会发现，**需求分析的深度不足**——用户故事流于表面，优先级矩阵没有说明判定依据，评审结论缺乏真实审批评议。

好消息是：MVP 范围本身是合理的，11 项功能对单人 4 周来说虽然紧但可行。问题出在需求分析过程和评审结论的质量上。如果以这份文档直接进入 PRD Lite，PRD 阶段会回头补需求分析该做的事（比如明确时段粒度、定义状态流转规则），反而更慢。

---

## B. 发现的问题清单

### B1. 需求分析深度不足（检查重点 1）

文档第 2 节写了 2.1 用户角色和 2.2 用户故事，但：

**用户故事只有正向场景，缺少异常与边界。**  
比如"预约提交"这个功能，只写了"作为一个学生，我想预约一个空闲座位并选择时段"，但没有提到：
- 如果选择的时段已经有人预约了怎么办？
- 如果选择的开始时间晚于结束时间怎么办？
- 如果选择的是过去的时段怎么办？
- 一次预约最长可以约多久？
- 一个人同时可以有多少个未完成的预约？

**这些边界情况在编码阶段一定会遇到，现在不定义，写代码时就靠开发者现场拍脑袋。** 需求分析的真正价值就在于提前把这些模糊地带暴露出来并达成共识。

**用户故事没有对应验收条件。**  
立项单里的成功标准以验收用例的形式写得很清楚，但到了需求分析文档，用户故事反而退化为纯描述。需求分析应该把每个用户故事展开为具体的验收条件（Acceptance Criteria），而不是简单重复用户故事模板。

### B2. 功能优先级矩阵缺少判定依据（检查重点 2）

文档 3.1 节列出了 15 项功能并标了优先级，但**没有说明为什么某个功能是 P0 而不是 P1**。

例如：
- F03 座位详情是 P1，但 F02 座位列表是 P0。——为什么？如果没有详情页，用户从列表直接点预约行不行？如果行，那详情页可以降级甚至取消，节省一个页面。
- F15 用户端首页是 P1，但它的内容（总座位数、空闲座位数）和管理端统计页 F12 有重叠。——首页的数据从哪里来？是和统计页调用同一个接口，还是各查各的？
- F14 预约冲突检测是 P0"大"工作量，但系统中最复杂的就是这段逻辑。为什么不分拆？先做基础提交（允许冲突，后靠管理员手动避免），再加冲突检测？

**优先级需要附带简单的判定理由**，例如：
> F03 座位详情（P1）：用户可以从座位列表直接预约，详情页非核心路径。如时间不足可合并到列表页的弹窗中。

### B3. 评审结论太"假"（检查重点 5）

第 5 节"需求评审结论"写了 4 条评审意见和一句"通过"，但：

- **谁参加的评审？** 没有列参加人。一个项目评审至少要写明参加角色（PM、开发、客户代表）。
- **具体的讨论和争议是什么？** 评审记录中没有任何异议、修改、妥协的过程。一次真实的评审不可能所有意见都是"合理"的，总有争议和取舍。
- **遗留事项写了 3 条，但评审结论却是"通过"。** 第 5.3 节说"预约时段的具体粒度需在 PRD 中明确"，这说明 MVP 范围里有一个关键的决策点还没定。不知道是按小时还是按半小时预约，页面设计、数据库设计、冲突检测逻辑都不一样。**这不是"遗留事项"，这是"前置条件"。** 应该在评审中要求补充，而不是放行到 PRD。

### B4. MVP 范围边界不够锋利（检查重点 3）

第 4 节说 MVP 包含 11 项功能，但 4 周做完 11 项功能（其中 F04、F14、F15 标注了"中/大"工作量），**没有提供一个可操作的"砍功能触发条件"。**

对比立项单中的里程碑优先级原则，它明确写了：
> 如果进度滞后，统计页可降级为纯数字卡片

但需求分析文档没有做同样的事。MVP 范围的 11 项功能缺少内部优先级排序。如果第 3 周结束时发现时间不够，该砍谁？

### B5. 最终范围适合单人项目，但风险没传递过来（检查重点 6）

11 项功能、4 周、1 个人，节奏合理，但问题是：

- 文档完全没有引用立项单中的风险（R1-R5）。立项单明确说了"4 周时间不够""页面数量可能失控""AI 代码不稳定"，但这些风险在需求分析中没有任何对应的应对措施。
- 比如针对 R1（时间不够），应该在 MVP 范围中明确"如果 3 周末尾用户端没闭环，管理端统计页砍掉"。但文档没有写。

### B6. 工作量评估缺乏依据

3.1 节的"工作量评估"列写了"小/中/大"，但没有说明每个级别的标准：
- "小"是多少小时？
- "大"又是多少？
- F14（预约冲突检测）和 F04（预约提交）合并标注为"大"，但这是两个功能的合并量还是每个都大？

## C. 必改项

### C1. 补充核心用户故事的异常场景验收条件

选择 3 个核心功能（预约提交、取消预约、座位管理中的删除座位），每个补充至少 2 条异常/边界验收条件。

以预约提交为例，补充到用户故事下方：

```
功能：预约提交（F04）

正常场景验收条件：
  AC-01: 学生选择一个空闲座位、选择未来时段（如 09:00-12:00）、点击提交 → 预约成功，状态为"预约中"
  AC-02: 学生在"我的预约"中看到刚刚提交的预约

异常场景验收条件：
  AC-03: 学生选择已被他人预约的时段 → 提示"该时段已被预约"，预约不成功
  AC-04: 学生选择开始时间晚于结束时间（如 14:00-09:00） → 提示"结束时间不能早于开始时间"
  AC-05: 学生选择过去的时间（如昨天） → 提示"不能预约过去的时间"
  AC-06: 学生已有 3 条"预约中"的记录，再次提交 → 提示"您已有 3 个未完成预约，请先取消或使用后再预约"
```

### C2. 评审结论需要重写，至少包含：

- 参加评审角色（示例：开发 1 人、PM 1 人）
- 至少 1 条实际讨论过的争议点及最终决定
- 遗留事项必须标注"是否阻塞 MVP 确认"，如果阻塞则评审结论应为"不通过，需补充后再次评审"

预约时段粒度（第 5.3 节第 1 条）属于阻塞项，不解决无法进入 PRD。

## D. 建议优化项

### D1. 优先级矩阵补充判定理由

对每一项 P0/P1，用一句话说明为什么是这个优先级。例如：

```
F15 用户端首页（P1，建议降为 P2）
理由：首页的"总座位数/空闲座位数"在座位列表页顶部也能展示，
不一定要独立一个页面。如果时间紧，直接让座位列表页兼做首页。
```

### D2. MVP 范围增设"渐进式降级路径"

模仿以下格式：

```
MVP 满载：11 项功能全部完成

如果第 3 周结束进度 < 70%：
  降级方案 A（推荐）：砍掉 F12 统计页，改为纯文字数字卡片嵌入座位管理页
  降级方案 B（备选）：砍掉 F03 座位详情页，预约入口直接放在座位列表

如果第 2 周结束用户端未闭环：
  降级方案 C：管理端只做预约管理，不做座位 CRUD，种子数据写死在数据库初始化中
```

### D3. 工作量评估改用粗略人时

把"小/中/大"改为估算范围：

| 级别 | 估算范围 | 示例 |
|------|----------|------|
| 小 | 2-4 小时 | F01 体验账号登录（控制器 + 简单视图） |
| 中 | 4-8 小时 | F02 座位列表（视图 + 查询 + 筛选） |
| 大 | 8-16 小时 | F04+F14 预约提交 + 冲突检测（表单验证 + 数据库校验 + 事务处理） |

这样能算出总工作量，判断 4 周是否真的够。

### D4. 需求分析文档交叉引用立项单中的风险

在 MVP 范围章节的末尾增加一段：

```
风险对应说明：
- 立项单风险 R1（时间不够）：已通过降级方案 D1/D2 应对
- 立项单风险 R2（范围膨胀）：MVP 范围锁定后非 P0 功能不纳入
- 立项单风险 R4（AI 代码不稳定）：预约冲突检测逻辑需人工审查
```

## E. 是否可以进入下一步 "PRD Lite"

| 条件 | 状态 |
|------|------|
| C1 补充核心用户故事的异常场景验收条件 | ❌ 未修改 |
| C2 评审结论重写，阻塞项标记清楚 | ❌ 未修改 |
| D1 优先级补充判定理由 | ⚠️ 建议采纳 |
| D2 MVP 补充降级路径 | ⚠️ 建议采纳 |
| D3 工作量改用估算范围 | ⚠️ 建议采纳 |
| D4 交叉引用风险 | 💡 可选 |

**结论：完成 C1 和 C2 后可进入 PRD Lite。**

C1 保证编码阶段不会因为边界情况反复改代码，C2 保证审批不是走过场。D 组建议项虽然不是强制阻塞，但强烈建议一并修改，因为对单人项目来说，降级路径和风险引用是实际保底的唯一机制——没有人帮你看范围，你自己得知道什么时候该停。
