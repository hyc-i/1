## 2. 项目背景

高校图书馆占座问题长期存在——学生到馆后发现没有空位，或者座位被人用书本长期占用。大部分学校目前靠人工巡查或纸质登记管理，效率低、体验差。

一个轻量的 Web 座位预约系统就可以解决这个问题：学生提前或到馆后选座预约，预约后获得该座位的临时使用权，离开时释放座位。相比纸质登记或第三方 App，学校自建系统可控、无广告、可定制。

本课程第三阶段模拟"入职后独立负责第一个小项目"，开发时长 4 周，最终交付可演示的 Web 系统。

## 3. 目标用户

| 角色 | 说明 |
|------|------|
| 学生（普通用户） | 需要预约图书馆座位的人，使用用户端功能：浏览座位、预约、取消、查看自己的预约记录 |
| 管理员 | 图书馆工作人员，使用管理端功能：管理座位（增删改）、查看所有预约、查看统计数据 |

**本期不开发注册流程**。用户端采用模拟登录（点击体验账号即可进入），管理端采用 Session / Cookie 轻量登录（预置管理员账号）。

---

## 4. 项目目标

用 4 周时间，开发一个基于 ASP.NET Core MVC 的图书馆座位预约系统 Web 应用，覆盖"选座→预约→管理→统计"的完整闭环，最终提交可运行的 GitHub/Gitee 仓库。

具体来说：

- 学生可以浏览图书馆座位布局，查看座位状态（空闲/已预约/使用中），预约空闲座位，取消自己的预约
- 管理员可以管理座位信息（添加、编辑、删除座位），查看和取消任意预约，查看基础统计（今日预约数、座位利用率等）
- 系统在启动时自动初始化演示数据（座位表 + 管理员账号），开箱即用

## 5. 本期必须完成内容（V1.0）

### 用户端（6 个页面）

| 编号 | 页面 | 说明 |
|------|------|------|
| U-01 | 用户首页 | 欢迎页，显示图书馆简介、座位总数、当前空闲座位数、快捷操作入口 |
| U-02 | 座位列表 | 以卡片或表格形式展示所有座位，标注座位编号、区域、状态（空闲/已预约/使用中），可按区域筛选 |
| U-03 | 座位详情 | 点击单个座位后查看详细信息（楼层、区域、座位号、设施描述），如果空闲则显示"立即预约"按钮 |
| U-04 | 预约提交 | 确认预约信息，选择预约时段（如 9:00-12:00），提交后生成预约记录 |
| U-05 | 我的预约 | 当前用户的所有预约记录列表，显示座位号、时段、状态（预约中/已签到/已取消/已完成） |
| U-06 | 取消预约 | 在我的预约页面点击取消，确认后预约状态变为"已取消"，座位释放 |

### 管理端（4 个页面）

| 编号 | 页面 | 说明 |
|------|------|------|
| A-01 | 管理员登录 | 输入账号密码登录管理后台 |
| A-02 | 预约管理 | 查看所有学生的预约记录，可按状态筛选，支持手动取消预约 |
| A-03 | 座位管理 | 座位 CRUD：添加新座位、编辑座位信息、删除座位（有预约的座位不可删除） |
| A-04 | 统计页 | 展示关键数据卡片：总座位数、今日预约数、座位利用率（百分比）、热门区域排名 |

### 数据与基础设施

- 数据库使用 SQL Server LocalDB，EF Core Code-First 方式建表
- 系统启动时自动初始化种子数据（约 30-50 个座位 + 1 个管理员账号）
- 用户端模拟登录：首页提供"体验学生 A""体验学生 B"等快捷入口，点击即切换身份
- 管理端 Session/Cookie 登录：预设用户名 `admin`，密码 `123456`

## 6. 本期暂不开发内容

以下功能明确排除在 V1.0 之外，不允许在 4 周内擅自追加：

- ❌ 支付功能（任何形式的预约押金或扣费）
- ❌ 扫码签到/签退（二维码扫描、NFC 等）
- ❌ 消息推送（短信、邮件、站内信、App 推送）
- ❌ 复杂账号体系（注册、找回密码、手机号绑定、第三方登录）
- ❌ 多馆区联动（本期只做一个馆/一个楼层的座位）
- ❌ 违规计分/黑名单机制
- ❌ 实时座位图（Canvas/ SVG 绘制的楼层平面图，本期只要表格/卡片列表）
- ❌ 小程序 / 移动 App / 响应式适配（本期仅保证桌面端 Chrome 浏览器可用）
- ❌ 多语言 / 主题切换

## 7. 页面规模与页面分布

**总计：10 个页面**

| 功能区 | 页面数 | 页面清单 |
|--------|--------|----------|
| 用户端 | 6 | 首页、座位列表、座位详情、预约提交、我的预约、取消预约（与我的预约同页） |
| 管理端 | 4 | 管理员登录、预约管理、座位管理、统计页 |
| 共用布局 | (1) | _Layout.cshtml（共享导航与样式框架） |

**单人 4 周评估**：10 个页面 ≈ 每周 2.5 个页面。考虑到 EF Core 能大幅减少数据访问代码、Bootstrap 提供现成 UI 组件，节奏合理。如果第 2 周结束时用户端 6 页还没完成，第 3 周需要压缩管理端页面规模。

## 8. 关键业务主流程

```
学生选座预约流程：
  学生点击体验账号登录
    → 进入用户首页（看到空闲座位数）
      → 点击"选座预约"进入座位列表
        → 筛选区域或直接浏览，查看座位状态
          → 点击空闲座位进入详情
            → 点击"立即预约"，选择时段
              → 提交预约 → 状态变为"已预约"
                → 在"我的预约"中可以看到该记录

管理员审核管理流程：
  管理员登录
    → 进入预约管理页（看到所有预约）
      → 可按状态筛选（预约中/已完成/已取消）
        → 可手动取消某个预约释放座位
    → 进入座位管理页
      → 新增/编辑/删除座位
    → 进入统计页
      → 查看今日预约数、座位利用率等数据
```

## 10. 风险与假设

### 风险

| 编号 | 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|------|----------|
| R1 | **4 周时间不够**：页面数量估算偏乐观，中间还可能被其他课程作业打断 | 中高 | 管理端 2 个页面做不完 | 提前固定优先级：用户端 > 管理端，如果第 3 周结束用户端没闭环，管理端只做预约管理 + 登录 |
| R2 | **页面数量失控**：写着写着觉得"再加一个功能也很好"（比如加个评论、加个收藏） | 高 | 范围膨胀，核心功能没做完 | 用第 6 节"暂不开发"清单做护栏，任何新增功能必须 PM 确认后方可加入 |
| R3 | **SQL Server LocalDB 安装/配置问题**：学生本机环境差异大，LocalDB 可能无法正常启动 | 中 | 浪费 1-2 天调试环境 | 提前确认 LocalDB 可用性，备选方案：若 LocalDB 不可用，降级至 SQLite + EF Core |
| R4 | **AI 生成代码不稳定**：使用 AI 辅助开发时，生成的代码可能包含低级 bug 或不一致的设计 | 高 | 调试时间超出预期 | 使用 AI 生成代码后必须逐段理解再提交，不要盲贴；关键逻辑（预约冲突检测、状态流转）人工把关 |
| R5 | **Git 操作失误**：误删文件、冲突不会解决、commit 信息混乱 | 中 | 项目文件损坏或无法提交 | 每个里程碑结束后拍照式 commit，第 1 周先练一次 Git 基本操作 |

### 假设

- 开发机已安装 Visual Studio 2022+（或 VS Code + .NET 8 SDK）、SQL Server LocalDB
- 项目使用 Chrome 浏览器演示，仅保证桌面端显示正常
- 最终通过 GitHub Desktop 或 git 命令行提交到远端仓库
- 演示时使用本地 `localhost` 访问，不需要公网部署

| 里程碑 | 时间 | 交付内容 | 验收条件 | 备注 |
