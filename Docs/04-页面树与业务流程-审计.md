## B. 发现的问题清单

### B1. 与 PRD Lite 的口径偏差未声明（检查重点 1）

PRD Lite 第 14 节明确说：
> "座位 CRUD（新增/编辑/删除）—— 种子数据预置 30+ 座位，演示足够" 归属 "本期不做，V1.1 考虑"

但本页面树文档第 8 节把 `PA-03 Views/Admin/Seats.cshtml` 列为"本期开发"，第 3 节描述其功能为"查看座位列表 + 新增/编辑/删除座位"。

这不是文档写错了——是口径本身变了。但文档中没有一句话说明"为什么 PRD 说不做但页面树里又加了"。如果有新人交接时只读了 PRD Lite 和这份文档，会发现前后矛盾。

**实际影响**：不会阻塞开发（页面数从 8 变成 9，还在可控范围），但需要提醒开发者：座位的"新增/编辑/删除"页面需要独立的视图和 POST 处理逻辑，工作量比 PRD 估算多了约 1-2 天。

### B2. "扩展功能"标注语义模糊（检查重点 6）

文档两处使用了"扩展功能"的说法：

- 第 220 行：`（管理座位 CRUD，本期为扩展功能）`
- 第 300 行：`座位 CRUD（扩展功能）`

"扩展功能"可以理解成：
- ① "这是一个额外功能，但本期做"（即属于范围）
- ② "这是一个可选项，做不完就砍掉"（即边缘功能）
- ③ "这是扩展包，本期不做"（即暂缓）

第 300 行的上下文（前置页面 ≠ 无，后置页面 = 自身刷新）暗示它确实在本期范围内，但"扩展功能"这个词容易让人以为它不强制。在单人项目中，"做不做"应该二选一，不应该有暧昧区域。

### B3. 座位管理页 CUD 三个操作未拆开描述

第 83 行把座位管理页的功能写成"查看座位列表 + 新增/编辑/删除座位"，但：
- "新增"和"编辑"通常需要表单页面或 Modal（额外的视图文件或 partial view）
- "删除"需要确认弹窗和 POST 请求
- 一个页面承载 4 个操作（列表 + 增 + 改 + 删），视图文件会很大

建议在页面树中明确这些操作是用 Modal 还是独立子页面，或者至少做个备注。否则下一阶段的 UI 设计可能会遗漏"新增座位表单"的布局。

### B4. 用户首页的双重状态描述清楚，但可以更聚焦

文档中"用户首页"在页面树中有两种形态（未登录、已登录），在第 2.3 节和第 4.2 节都做了详细描述。描述本身没问题，但入口路径的默认行为没有说清楚：

- 用户访问 `/` 时，如果 Session/Cookie 中有身份标记，是直接进入"已登录首页"还是仍然显示"未登录首页"让用户再点一次？
- 如果是前者（自动登录），那么"退出"后清除标记，首页回到未登录状态——这是合理的
- 建议在第 2.2 节补充一句："如果 Session/Cookie 中已有身份标记，自动进入已登录首页"

## D. 建议优化项

### D1. 声明"座位管理页"的口径变化

在第 3 节（管理端页面树）或第 8 节（当前范围内页面）中增加一条批注：

```
> 注意：本页面在 PRD Lite 中列为 V1.1 范围，经最终口径确认调整为本期开发。
> 涉及工作量：约需额外 1-2 天完成新增/编辑/删除 3 个操作的视图和 POST 处理。
```

### D2. 替换"扩展功能"为更明确的标签

将第 220 行和第 300 行中的"扩展功能"改为"本期开发"或"本期范围"，消除歧义。

### D3. 补充"新增/编辑座位"的 UI 承载方式

在座位管理页 PA-03 的描述中补充：

```
座位管理页的"新增"和"编辑"操作建议使用 Bootstrap Modal 弹窗承载表单，
不新增独立子页面。删除操作使用确认弹窗 + POST 提交。
```
