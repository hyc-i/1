# 数据库设计 · 图书馆座位预约系统

---

## 1. 数据设计目标

| 目标 | 说明 |
|------|------|
| 刚好够用 | 只建本期迭代需要的表，不预留给未来功能预留字段 |
| 查询优先 | 索引设计覆盖所有筛选和统计查询路径，不出现全表扫描 |
| 一致可靠 | 唯一索引防重复，业务层防重叠，不可逆状态变更由代码保证 |
| 种子可用 | 启动即初始化 30~50 条座位 + 1 条管理员，无需手动 SQL |

---

## 2. 核心实体清单

| # | 实体 | 表名 | 角色 | 数据来源 |
|---|------|------|------|----------|
| 1 | **Seat** | Seats | 座位档案（区域、楼层、设施描述） | 种子数据预置 |
| 2 | **Reservation** | Reservations | 预约记录（谁在什么时间约了哪个座位） | 学生提交生成 |
| 3 | **Admin** | Admins | 管理员账号（用户名 + 密码哈希） | 种子数据预置 |
| — | 体验学生 | — | 不建表，身份仅存在 Session 中 | 前端 3 个固定按钮 |

**体验学生不建表的理由**：3 个体验账号（A/B/C）是演示用简化方案，不需要注册流程、不需要密码、不需要持久化。身份仅存于 `Session["StudentName"]` 和 `Session["StudentIdentifier"]`，页面刷新或切换到其他体验账号即切换身份。这是需求分析阶段确认的设计决策。

## 3. 关系说明

### 3.1 ER 图（文字版）

```
  ┌─────────────┐          ┌──────────────────┐
  │    Seat     │          │   Reservation    │
  ├─────────────┤          ├──────────────────┤
  │ Id (PK)     │──1:N────>│ SeatId (FK)      │
  │ SeatNumber  │          │ Id (PK)          │
  │ Area        │          │ StudentName      │
  │ Floor       │          │ StudentIdentifier│
  │ Description │          │ ReservationDate  │
  │ IsActive    │          │ StartTime        │
  └─────────────┘          │ EndTime          │
                           │ Status           │
                           │ CreatedAt        │
  ┌─────────────┐          │ CancelledAt      │
  │   Admin     │          └──────────────────┘
  ├─────────────┤
  │ Id (PK)     │    说明：
  │ Username    │    · Seat 与 Reservation 是 1:N 关系
  │ PasswordHash│    · Admin 与其他表无外键关系（独立认证表）

```

### 3.2 关系汇总

| 父实体 | 子实体 | 关系类型 | 外键 | 级联删除 |
|--------|--------|----------|------|----------|
| Seat | Reservation | 1:N | Reservation.SeatId → Seat.Id | Restrict（禁止级联删除） |
| Admin | — | 独立表 | 无外键 | — |

**Restrict 的含义**：如果某个 Seat 还有预约记录关联，不允许删除该 Seat，必须先处理完关联的预约或手动将 Seat.IsActive 置为 false（软停用）。

## 4. 表结构设计

### 4.1 Seats（座位表）

**物理表名**：`Seats`
**数据量预估**：种子 30~50 条，增长极低（仅管理员手动增删）

| # | 字段名 | 类型 | 约束 | 默认值 | 说明 |
|---|--------|------|------|--------|------|
| 1 | Id | int | PK, 自增 | — | 主键 |
| 2 | SeatNumber | nvarchar(20) | NOT NULL, UNIQUE | — | 座位编号，如 "A-12"。业务唯一标识 |
| 3 | Area | nvarchar(50) | NOT NULL | — | 区域名称，如 "三楼自习区" |
| 4 | Floor | nvarchar(20) | NULL | — | 楼层，如 "3F" |
| 5 | Description | nvarchar(200) | NULL | — | 设施描述，如 "靠窗、有插座" |
| 6 | IsActive | bit | NOT NULL | 1 (true) | 是否启用。false 表示该座位已停用，不在列表中显示 |

**索引**：`IX_Seats_SeatNumber`（唯一索引，`SeatNumber`）
**种子数据**：3~4 个区域，每个区域 8~15 个座位，合计 30~50 条

### 4.2 Reservations（预约记录表）

**物理表名**：`Reservations`
**数据量预估**：演示期间数十条，长期使用每年约数千条

| # | 字段名 | 类型 | 约束 | 默认值 | 说明 |
|---|--------|------|------|--------|------|
| 1 | Id | int | PK, 自增 | — | 主键 |
| 2 | SeatId | int | FK → Seats.Id, NOT NULL | — | 关联座位 |
| 3 | StudentName | nvarchar(50) | NOT NULL | — | 学生显示名，如 "学生 A" |
| 4 | StudentIdentifier | nvarchar(50) | NOT NULL | — | 学生标识，如 "student_a"。用于按学生筛选 |
| 5 | ReservationDate | date | NOT NULL | — | 预约日期，只存日期部分（2026-07-07） |
| 6 | StartTime | datetime2(0) | NOT NULL | — | 预约开始时间（含日期），如 2026-07-07 09:00 |
| 7 | EndTime | datetime2(0) | NOT NULL | — | 预约结束时间（含日期），如 2026-07-07 11:00 |
| 8 | Status | nvarchar(20) | NOT NULL | '预约中' | 状态：预约中 / 已取消 |
| 9 | CreatedAt | datetime2(0) | NOT NULL | GETUTCDATE() | 创建时间 |
| 10 | CancelledAt | datetime2(0) | NULL | — | 取消时间，Status = '已取消' 时有值 |

**索引**：

| 索引名 | 字段 | 类型 | 用途 |
|--------|------|------|------|
| PK_Reservations | Id | 聚集 | 主键 |
| IX_Reservation_Seat_Time | SeatId, ReservationDate, StartTime, EndTime | **非唯一** | 加速冲突检测查询 |
| IX_Reservation_Student | StudentIdentifier, ReservationDate | 非唯一 | 加速"我的预约"列表查询 |
| IX_Reservation_Status | Status, ReservationDate | 非唯一 | 加速管理端按状态筛选 |
| IX_Reservation_Date | ReservationDate | 非唯一 | 加速统计页按日期聚合 |

### 4.3 Admins（管理员表）

**物理表名**：`Admins`
**数据量预估**：1 条（种子数据）

| # | 字段名 | 类型 | 约束 | 默认值 | 说明 |
|---|--------|------|------|--------|------|
| 1 | Id | int | PK, 自增 | — | 主键 |
| 2 | Username | nvarchar(50) | NOT NULL, UNIQUE | — | 管理员用户名，种子写入 "admin" |
| 3 | PasswordHash | nvarchar(200) | NOT NULL | — | 密码哈希值，存 `BCrypt.Net` 或 `ASP.NET Core Identity` 兼容哈希 |

**索引**：`IX_Admins_Username`（唯一索引，`Username`）

## 5. 关键字段说明

### 5.1 SeatNumber 命名规范

种子数据建议的区域前缀和编号规则：

| 区域 | 前缀 | 编号范围 | 示例 |
|------|------|----------|------|
| 一楼大厅 | C | C-01 ~ C-10 | C-05 |
| 二楼阅览区 | B | B-01 ~ B-15 | B-12 |
| 三楼自习区 | A | A-01 ~ A-15 | A-08 |

编号不要求连续（删除座位后不补号），但确保 `SeatNumber` 全局唯一。

### 5.2 StudentIdentifier 取值

| 体验账号 | StudentName | StudentIdentifier |
|----------|-------------|-------------------|
| 学生 A | 学生 A | student_a |
| 学生 B | 学生 B | student_b |
| 学生 C | 学生 C | student_c |

此数据硬编码在 `HomeController` 或 `StudentContext` 中，不在数据库里。

### 5.3 时间的存储规则

**StartTime / EndTime 的特殊设计**：
这两个字段虽然是 `datetime2` 类型，但在业务层面可以拆解为"日期部分 + 时间部分"的复合含义。存储时，日期部分应等于 `ReservationDate` 对应的日期。例如：

- 预约日期：2026-07-07
- 时段：09:00 ~ 11:00
- StartTime = 2026-07-07 09:00
- EndTime = 2026-07-07 11:00

这样设计的好处：
1. 冲突检测可以直接比较 `StartTime < @EndTime AND EndTime > @StartTime`，不需要拆分日期和时间
2. 排序简单：直接按 `StartTime` 倒序即可
3. 跨日期预约（如 22:00 ~ 02:00）虽不支持，但本系统不涉及跨日期场景

### 5.4 ReservationDate 字段

虽然 `StartTime` 包含日期信息，但单独保留 `ReservationDate` 字段是为了：

| 原因 | 说明 |
|------|------|
| 查询效率 | `WHERE ReservationDate = @today` 比 `WHERE CAST(StartTime AS DATE) = @today` 高效，且可用索引 |
| 统计聚合 | 统计页按日期分组时直接使用 date 类型字段 |
| 语义清晰 | 代码中 `r.ReservationDate` 明确表示"预约的日期"，读者不需要猜测 |

### 5.5 PasswordHash

本项目不使用 ASP.NET Core Identity。管理员密码验证用 `PasswordHasher<TUser>`（Microsoft.AspNetCore.Identity 命名空间中的独立类，可脱离 Identity 框架单独使用）或 BCrypt.Net。

```csharp
// 用 PasswordHasher<Admin>（ASP.NET Core 内置，不依赖 Identity 框架）
var hasher = new PasswordHasher<Admin>();
string hash = hasher.HashPassword(admin, "123456");

// 验证时
var result = hasher.VerifyHashedPassword(admin, storedHash, inputPassword);
// result == PasswordVerificationResult.Success 表示密码正确
```

## 6. 状态字段设计

### 6.1 Reservations.Status

| 状态值 | 含义 | 初始/最终 | 可转换到 |
|--------|------|-----------|----------|
| `预约中` | 预约有效，座位被占用 | 初始状态 | `已取消` |
| `已取消` | 预约已手动取消，座位释放 | 最终状态，不可逆 | （无） |

**状态转换规则**：
- 新建预约时 `Status = '预约中'`
- 学生或管理员手动取消时 `Status = '已取消'`，同时写入 `CancelledAt`
- 已取消的记录不可恢复为 `预约中`
- V1.0 不做自动过期（不实现 `已使用` / `已过期` 等状态）

**为什么用字符串而不是 int 或 enum**：
- 字符串 `预约中` / `已取消` 在数据库里可读，调试时不需要查映射表
- EF Core 存 enum 默认存 int，查询时 `WHERE Status = 0` 不直观
- 只有 2 个值，没有性能问题

### 6.2 座位状态（虚拟字段，不存储）

座位状态**不是数据库字段**，而是通过查询 `Reservations` 表实时计算的逻辑状态：

| 逻辑状态 | 计算条件 | 代码实现 |
|----------|----------|----------|
| `空闲` | 该座位当天没有状态为`预约中`且时段覆盖当前时间的预约记录 | `!Reservations.Any(r => r.SeatId == seatId && r.ReservationDate == today && r.Status == "预约中" && r.StartTime < now && r.EndTime > now)` |
| `已预约` | 该座位当天至少有一条状态为`预约中`且时段覆盖当前时间的预约记录 | 上述条件的否定 |

**注意**：座位列表页展示状态时，是查询"当前时间点"的实时状态。如果用户查看未来日期的列表，未来日期所有座位都显示"空闲"（因为当天还没有预约记录）。预约提交时才做精确的时段冲突检测。

## 7. 查询与筛选字段设计

### 7.1 查询路径与对应索引

| 查询场景 | WHERE 条件 | 推荐索引 | 说明 |
|----------|-----------|----------|------|
| 按区域筛选座位 | `Seat.Area = @area AND Seat.IsActive = true` | —（数据量小，30~50 条不需索引） | 全表扫描可接受 |
| 座位详情 | `Seat.Id = @id` | PK (聚集索引) | 主键查询 |
| 冲突检测 | `Reservation WHERE SeatId=@sid AND ReservationDate=@d AND Status='预约中' AND StartTime<@end AND EndTime>@start` | `IX_Reservation_Seat_Time` | 最关键的查询，应用层必走 |
| 我的预约列表 | `Reservation WHERE StudentIdentifier=@sid ORDER BY CreatedAt DESC` | `IX_Reservation_Student` | 按学生标识 + 日期排序 |
| 管理端按状态筛选 | `Reservation WHERE Status=@s ORDER BY CreatedAt DESC` | `IX_Reservation_Status` | 覆盖"预约中"/"已取消"筛选 |
| 统计页今日预约数 | `Reservation WHERE ReservationDate=@today AND Status='预约中'` | `IX_Reservation_Date` | 日期筛选 |
| 统计页利用率 | `Reservation WHERE ReservationDate=@today AND Status='预约中'` + DISTINCT SeatId | `IX_Reservation_Date` | 同上索引可覆盖 |
| 统计页热门区域 | `Reservation JOIN Seat WHERE ReservationDate=@today AND Status='预约中' GROUP BY Area` | `IX_Reservation_Date` + `IX_Reservation_Seat_Time` | 需要通过 SeatId 关联到 Area |

### 7.2 不会出现的查询模式

| 查询模式 | 不做的原因 | 替代方案 |
|----------|-----------|----------|
| 模糊搜索座位描述 | 本系统没有搜索功能 | 下拉框区域筛选即可 |
| 按管理员账号搜索预约 | 只有一个管理员 | 不实现 |
| 跨多天范围查询预约 | 统计页仅按"今天"统计 | 本期不实现日期范围选择器 |

### 7.3 关于唯一索引无法防御时段重叠的说明

`IX_Reservation_Seat_Time` 是**非唯一索引**，它的作用是加速冲突检测查询，**不能**用唯一约束来防御时段重叠。

原因解释：

```
已有预约：2026-07-07 09:00 ~ 12:00（Seat A-01）
新的预约：2026-07-07 10:00 ~ 11:00（Seat A-01）

两条记录的 (SeatId, ReservationDate, StartTime, EndTime)：
  现有: (1, 2026-07-07, 2026-07-07 09:00, 2026-07-07 12:00)
  新增: (1, 2026-07-07, 2026-07-07 10:00, 2026-07-07 11:00)

这四条字段值在两条记录中完全不同，唯一索引不会阻止写入。
```

因此，冲突检测必须由**应用程序业务层**（`ReservationService.CreateReservationAsync`）在写入前先用 SELECT 查询确认无重叠记录。这是不可绕过的设计决策。

## 8. 统计字段与统计口径

### 8.1 4 个统计指标的定义

| 指标 | 字段来源 | 计算方法 | 展示位置 |
|------|----------|----------|----------|
| 总座位数 | Seats 表 | `SELECT COUNT(*) FROM Seats WHERE IsActive = 1` | 首页 + 统计页 |
| 今日预约数 | Reservations 表 | `SELECT COUNT(*) FROM Reservations WHERE ReservationDate = @today AND Status = '预约中'` | 统计页 |
| 座位利用率 | Reservations + Seats | `(当天有预约的座位数 / 总座位数) × 100%`<br>`当天有预约的座位数 = SELECT COUNT(DISTINCT SeatId) FROM Reservations WHERE ReservationDate = @today AND Status = '预约中'` | 统计页 |
| 热门区域 | Reservations JOIN Seats | `SELECT TOP 1 s.Area FROM Reservations r JOIN Seats s ON r.SeatId = s.Id WHERE r.ReservationDate = @today AND r.Status = '预约中' GROUP BY s.Area ORDER BY COUNT(*) DESC` | 统计页 |

### 8.2 首页统计口径（与统计页区别）

首页展示的 3 个数字（总座位数 / 空闲座位数 / 已预约数）与统计页的 4 个数字口径不同：

| 首页字段 | 口径 | 与统计页差异 |
|----------|------|-------------|
| 总座位数 | 同统计页 | 相同 |
| 空闲座位数 | `总座位数 - 当前有预约的座位数` | **实时**计算，基于"当前时间"的预约状态 |
| 已预约数 | `当前有预约的座位数` | **实时**计算，基于"当前时间"的预约状态 |

统计页的"今日预约数"是**当天提交的预约总量**（一个座位今天被预约了 3 次就算 3 次），而首页的"已预约数"是**当前被占用的座位数**（一个座位只算 1 次）。

### 8.3 聚合查询策略

4 个统计指标在一次数据库交互中完成：

```csharp
var today = DateTime.Today;

// 一次性查询所有需要的数据
var totalSeats = await _context.Seats.CountAsync(s => s.IsActive);

var todayActiveReservations = await _context.Reservations
    .Where(r => r.ReservationDate == today && r.Status == "预约中")
    .ToListAsync();  // 数据量小，全部拉内存再聚合

var todayCount = todayActiveReservations.Count;
var utilizedSeatCount = todayActiveReservations.Select(r => r.SeatId).Distinct().Count();
var utilizationRate = totalSeats > 0
    ? Math.Round((double)utilizedSeatCount / totalSeats * 100, 0)
    : 0;

// 热门区域需要 JOIN Seat 获取 Area
var hotArea = await _context.Reservations
    .Where(r => r.ReservationDate == today && r.Status == "预约中")
    .Join(_context.Seats, r => r.SeatId, s => s.Id, (r, s) => s.Area)
    .GroupBy(a => a)
    .OrderByDescending(g => g.Count())
    .Select(g => g.Key)
    .FirstOrDefaultAsync() ?? "暂无数据";
```

## 9. 数据一致性与约束说明

### 9.1 数据库层约束

| 约束 | 所在表 | 类型 | 作用 |
|------|--------|------|------|
| SeatNumber 唯一 | Seats | 唯一索引 | 不允许两个座位有相同编号 |
| Username 唯一 | Admins | 唯一索引 | 不允许两个管理员有相同用户名 |
| SeatId 外键 | Reservations | 外键 (Restrict) | 禁止删除有预约关联的座位 |
| ReservationDate 类型 | Reservations | `date` 类型 | 确保只存日期不存时间 |

### 9.2 应用层约束（数据库不强制，业务代码保证）

| 约束 | 说明 | 违反后果 |
|------|------|----------|
| 时段重叠检测 | 同一座位同一天同一时段不可有两条`预约中`记录 | 重复预约 |
| 状态不可逆 | `已取消`不可变为`预约中` | 数据逻辑混乱 |
| 取消归属校验 | 学生只能取消自己的预约 | 越权取消 |
| 结束时间 > 开始时间 | EndTime 必须晚于 StartTime | 时段数据无效 |
| 未来时间验证 | 不能预约过去的时间 | 历史数据被篡改 |
| 同一学生同一时段冲突 | 同一学生不能在同一时段预约两个不同座位 | 资源浪费 |

### 9.3 为什么部分约束放在应用层

时段重叠检测无法用数据库唯一索引实现（原因见 7.3 节），这是数据库模型本身的限制（关系型数据库不支持"排他约束"）。`IX_Reservation_Seat_Time` 作为非唯一索引只为加速查询，不能替代业务层逻辑。

### 9.4 外键 DeleteBehavior 选择

`Reservation → Seat` 的外键使用 `DeleteBehavior.Restrict`，意味着：
- 删除一个座位时，如果它有未删除的预约记录，数据库会抛出外键违规错误
- 代码中应先检查 `Seat.Reservations.Any(r => r.Status == "预约中")`，如果存在则禁止删除
- 推荐做法：将 `Seat.IsActive = false` 实现"软停用"，而不是物理删除

## 10. 当前阶段实现边界

### 10.1 本期建的表

| 表 | 是否新建 | 说明 |

| Seats | ✅ 新建 | 座位档案 |

### 10.2 本期不建的表

| 表名 | 不建的理由 |
|------|-----------|

| ReservationLogs（操作日志表） | 本期无审计需求，取消操作直接在 Reservations 表写 CancelledAt |

## 11. 默认补足项 / 当前假设

前序文档（03-PRD-Lite、04-页面树、07-系统设计说明）已给出完整的数据骨架：
- 3 张数据表、字段定义、索引、状态流转全部有明确描述
- 体验学生不建表、座位状态实时计算等设计决策已有明确理由

因此**不需要补充默认实体，未拆分任何已有字段结构**。以下仅列出设计过程中确认的补充约定：

### 11.1 补充约定清单

| # | 约定点 | 来源 | 设计选择 |
|---|--------|------|----------|
| 1 | Area 字段存储方式 | PRD 03 定义为 `string(50)` | 直接存在 Seats 表，不拆分 SeatAreas 表。因为本期仅记录区域名称，不需要区域的其他属性 |
| 2 | StartTime/EndTime 存储精度 | PRD 03 定义为 DateTime | 使用 `datetime2(0)`，精度到秒。不存毫秒（无意义） |
| 3 | 密码哈希方案 | PRD 03 要求非明文 | 使用 `PasswordHasher<Admin>`（ASP.NET Core 内置，可脱离 Identity 使用）或 BCrypt.Net |
| 4 | 唯一索引的局限性 | PRD 03 索引描述需修正 | PRD 03 第 11.3 节称"唯一组合索引防止重复预约"，此说法有误导。实际唯一索引只能防止完全相同的 (SeatId, ReservationDate, StartTime, EndTime) 组合，无法防止时段重叠（如 09:00~12:00 与 10:00~11:00 重叠但四个字段值都不同）。冲突检测必须依赖业务层代码 |
| 5 | 统计口径区分 | 07 系统设计附录 C | 首页"已预约数"= 当前被占用的座位数（实时）；统计页"今日预约数"= 当天预约总量（累计） |
| 6 | 种子数据构造策略 | 07 系统设计 3.1 | 使用 `HasData()` 或 `SaveChangesAsync()` 在首次启动时写入，非 SQL 脚本 |
| 7 | 软删除座位 | 07 系统设计 10.2 | 删除座位时设置 `IsActive = false` 而非物理删除，避免外键约束冲突 |

### 11.2 已被排除的潜在拆分方案

以下拆分方案在前序文档评审过程中被讨论并确认排除：

| 潜在方案 | 排除理由 |
|----------|----------|
| 将 Area 拆为独立表（SeatAreas） | 区域只有一个名称属性，拆表需要 JOIN 后 Area 信息才能展示，增加查询复杂度但收益为零 |
| 将 TimeSlot 拆为独立预约时段表 | 时段由用户自由选择，本系统不预置固定时段（如 09:00~12:00、14:00~17:00），不需要时段主数据 |
| 将学生信息建表（Users） | 3 个固定身份的演示系统，不值得建表。Session 方案代码量 < 20 行 |
| 将取消操作写入独立日志表（CancellationLogs） | 本期无审计需求，`CancelledAt` 字段足够记录取消时间。未来如需操作审计再加日志表 |
