# 项目立项单 · 审计报告

审计对象：Docs/01-项目立项单.md  
审计日期：2026-07-07  
审计结论：**不通过**，存在 3 项必改问题，修改后可进入下一步。

---

## A. 总体判断

这是一份**骨架完整但血肉不足**的立项单。格式上具备立项单的基本要素（背景、目标、范围、里程碑、风险），但**每个要素都存在不同程度的模糊和缺失**。作为单人课堂项目，方向正确，但细节远不够落地。如果拿着这份立项单直接开始开发，中期大概率出现范围蔓延或交付标准扯皮。

核心问题集中在三个方面：**目标无量化**、**成功标准不可演示**、**风险部分流于形式**。

---

## B. 发现的问题清单

### B1. 项目目标不够清楚

原文第 3 节：
> "实现活动记录的在线提交、审核、统计等基本功能。"

**问题**："基本功能" 是模糊词。对开发者来说，"基本" 可以是 3 个页面，对客户来说 "基本" 可能包含报表导出和数据分析。目标应该写 "完成什么、做到什么程度"。

同一个句子还把 "提交、审核、统计" 并列，但统计在整个文档中只在范围里提了一句，没有具体到统计什么指标。而在实际代码中 `DashboardStats` 模型包含 8 个字段，比如总学生数、总记录数、通过率等——这些应该提前写进目标而不是写代码时才发现。

### B2. 本期范围与暂不开发内容边界清楚，但仍有遗漏

**做得好的**：第 5 节 "暂不开发" 列出了 7 项，比较完整，尤其 "微信小程序/移动端" 和 "院系二级管理" 明确切出了边界。

**做得不够的**：

1. **第 4 节 "活动登记记录的提交、编辑、查看" 没有提到审核流程中的拒绝/驳回**。实际项目中审核状态有 "待审核→已通过" 和 "待审核→未通过" 两个方向，这是业务流程不可缺少的一半，范围中只字未提。
2. **未提及 "导出/打印"**。实际项目 `ActivityRecordsController` 中已经有 `ExportToExcel` 方法，说明导出是实际需要的功能，但范围和暂不开发中都没写。
3. **未提及 "审核日志/历史"**。实际代码设计了 `ReviewLogs` 表，但立项单里完全没覆盖这条线。

### B3. 页面规模基本适合单人课堂项目

11 个页面 = 1 人 × 8 周，平均每周 1.4 个页面，加上控制器和数据层，节奏合理。但：

- **缺少 "导入结果页"**：第 11 项 "学生信息导入页" 只写了一个页面，实际流程需要 2 个（导入表单 + 导入结果显示），这在 `Students/Import.cshtml` 中可见。
- **缺少 "无数据/空状态/错误页"**：异常处理页 (`Error.cshtml`) 已经存在但未列入页面清单。

### B4. 成功标准不够具体，不可演示、不可检查

原文第 8 节，5 条成功标准全是类似这样的：
> "学生能成功登录并查看自己的活动记录"

**问题**：这句话没有定义什么叫 "成功"。是输对密码就叫成功？输错密码应该怎样？"查看自己的活动记录" 看到什么程度——只看列表还是能点进去看详情？

以第 8 节的第 3 条为例，"辅导员能审核并给分" 没有指定：
- 审核后状态变成什么
- 通过给多少分、不通过给多少分
- 审核意见写在哪里
- 同一条记录能否重复审核

**正确写法应该像验收用例，每一条都是一个可以演示的操作步骤**。

### B5. 风险与假设不真实，缺少应对方案

第 9 节共列出 4 项：
1. "假设用户会使用浏览器访问系统"——这不算风险，Web 系统的默认前提
2. "假设学校网络环境稳定"——太泛，什么级别才算稳定？
3. "风险：开发时间不足"——没有具体原因，也没有写延期多少天算风险发生
4. "风险：需求变更"——同样没有写变更范围多大算触发

**缺失的关键风险**：
- **技术风险**：团队 1 人，对 ASP.NET Core MVC 熟练程度未知。如果卡在某个技术点上（比如身份认证、文件上传），项目直接停摆。这是单人项目最大的真实风险。
- **数据风险**：SQLite 在高并发场景下性能不足，如果真实上线使用，需要考虑迁移到 SQL Server。
- **依赖风险**：是否依赖学校现有系统（如教务系统、统一身份认证）获取学生数据？如果没有这些接口，手动维护学生信息的工作量多大？

### B6. 里程碑过多、粒度不一致

8 周 8 个里程碑，平均一周一个，数量可以接受，但**粒度不均匀**：

| 里程碑 | 问题 |
|--------|------|
| M1 项目启动 | 合理 |
| M2 数据库设计 | 第 2 周才设计数据库？通常和 M1 合并。而且交付物只写了 "建表脚本"，缺少数据字典 |
| M3 学生管理 | 第 3 周才做 CRUD，但 M4 的记录管理依赖学生数据，M3 滞后会影响 M4 |
| M4-M6 | 三个功能里程碑并行风险大——每个都涉及页面 + 控制器 + 数据层，实际工作量不均衡 |
| M7 测试 | 仅 1 周测试周期，对单人项目偏紧。而且没有区分单元测试、集成测试、验收测试 |
| M8 部署 | 缺少部署环境说明（服务器配置、域名、HTTPS） |

另一个问题：**里程碑之间没有依赖关系图**。M3（学生管理）不完成，M4（记录管理）无法开始，M5（学生门户）也依赖 M4。但文档是按时间顺序排列的，没有说清楚哪些可以并行、哪些必须串行。

## C. 必改项（不修改不能进入下一步）

### C1. 成功标准必须改为 "验收用例" 形式

每一条成功标准改为：
```
[编号] [操作角色] [操作] [输入] [预期结果]
```

示例：
```
SC-1 学生登录
  操作：学生在登录页输入学号和手机号，点击登录
  数据：学号 2024010101，手机号 13800000001
  预期：页面跳转到学生个人门户，顶部显示该生姓名

SC-2 学生提交活动记录
  操作：学生在创建页填写活动信息，点击提交
  数据：类别=志愿服务，级别=校级，时长=3.0
  预期：列表页出现新记录，状态="待审核"

SC-3 辅导员审核通过
  操作：管理员在审核页选择"已通过"，填写学分 2.0，点击保存
  预期：记录状态变更为"已通过"，学分 = 2.0，详情页可见审核历史
```

至少写 8-10 条，覆盖核心流程 + 边界情况（如驳回、重复提交、空数据）。

### C2. 风险部分必须补充真实风险及应对方案

至少补充：
```
R1 单人项目技术卡点风险
  描述：开发者对 ASP.NET Core MVC 掌握不足，身份认证/文件上传等功能耗时超预期
  概率：中高
  影响：进度延迟 1-2 周
  应对：提前准备学习资料，核心功能优先开发，预留缓冲时间

R2 学生数据来源不确定风险
  描述：系统所需学生数据需手工录入或从 Excel 导入，实际获取路径未确认
  概率：中
  影响：学生管理模块开发方向可能变化
  应对：确定最小可行方案——支持 Excel 导入，预留批量导入接口

R3 SQLite 并发限制风险
  描述：SQLite 不支持高并发写，多人同时提交审核可能导致写入失败
  概率：低（课堂演示场景）
  影响：演示时卡顿
  应对：课堂演示场景可接受；上生产前迁移至 SQL Server 或 PostgreSQL
```

### C3. 里程碑必须补充依赖关系和验收物

当前里程碑写得太简单，每个里程碑补充：
```
M3 学生管理
  前置依赖：M2（数据库建表完成）
  交付物：学生列表页、新增页、编辑页、删除功能、导入功能
  验收标准：
    - 可在列表页看到 125+ 条种子学生数据
    - 可按学号搜索学生
    - 可新增学生并显示在列表中
    - 可编辑学生信息
    - 可从 Excel 导入学生
    - 删除学生时有确认提示
```

## D. 建议优化项（提高质量，但不阻塞进入下一步）

### D1. 项目目标补充量化指标

```
当前：实现活动记录的在线提交、审核、统计等基本功能
建议：实现第二课堂活动记录的在线提交、审核授分、审核历史追溯与学分统计，
      目标支持 500 名在校生和 20 名辅导员/管理员同时使用
```

### D2. 范围中补充遗漏功能

- 审核流程中的驳回/不通过逻辑
- 审核历史日志的查看
- 记录导出功能
- 系统启动时自动初始化种子数据

### D3. 页面清单按角色分组呈现

```
所有用户通用（1 页）：登录页
管理员（8 页）：统计面板、学生列表、学生新增/编辑、学生导入、记录列表、记录创建、记录详情/审核
学生（2 页）：记录列表、记录创建
```

这样能直观看到每个角色需要多少页面，也便于分工或安排开发顺序。

### D4. 补充非功能性需求

- 支持最低并发数（课堂演示 ≥ 20 人同时操作）
- 页面响应时间（列表页 ≤ 2 秒）
- 浏览器兼容性（Chrome / Edge 即可）
- 密码安全（Hash 存储）

### D5. 增加一句话的 "退出标准"

例如："如果 4 周内无法完成学生管理模块的 CRUD，项目取消或降级为纯前端原型。"——这能防止在不可行的方向上浪费过多时间。
