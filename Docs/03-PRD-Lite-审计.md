## A. 总体判断

这是一份**质量不错的 PRD Lite**。结构完整、功能清晰、验收标准可操作。相比于上一阶段的 MVP 确认文档，这份 PRD 在"可执行性"上明显进步——每个功能都给出了最小验收标准，页面清单与实际视图文件对应，状态流转和冲突检测规则写得清楚。

主要问题集中在 **1 个必改的功能性缺陷**：预约记录的过期状态未处理，会导致演示时第二天座位全部显示为"已预约"。这在 MVP 确认时已经被允许（"暂不处理"），但在 PRD 中必须明确解决方案，否则写代码时会直接复制这个缺陷。

## B. 发现的问题清单

### B1. 【功能缺陷】预约记录无过期自动释放机制（检查重点 5）

文档第 12.2 节说"V1.0 不实现自动过期/签到，预约到期后需管理员手动取消"，同时在座位状态计算规则（12.1 节）说"查询 Reservations 表中当前时段是否有 '预约中' 的记录"。

**问题在于：这两个规则自相矛盾。** 如果 V1.0 不做自动过期，那么昨天 09:00-12:00 的预约在今天仍然是"预约中"，这条记录永远不会变为"已取消"或"已完成"。第二天所有座位的"当前时段查询"会返回"已预约"，整个系统就废了。

具体场景推演：

```
第 1 天：学生 A 预约座位 A-01，时段 09:00-12:00，状态"预约中" → 座位显示"已预约" ✅
第 2 天：任何学生打开系统 → 座位 A-01 仍显示"已预约"（因为同一座位、不同日期但系统中没有日期过滤逻辑）❌
```

这是**当前设计中最大的一个窟窿**。不补上，第 4 周演示时会直接翻车。

### B2. 验收标准缺少空状态场景（检查重点 3）

20 条验收标准覆盖了正常流程和异常输入，但没有一条描述"数据为空时应该显示什么"：

- AC-10："我的预约"页面时——如果当前学生一条预约都没有，应该显示空状态提示还是空白表格？
- AC-16：预约管理页——如果一条预约记录都没有，应该显示什么？
- AC-19：统计页——如果今天没有预约，利用率为 0%，是显示"0%"还是"暂无数据"？

这会在写代码时造成不一致：开发者 A 可能显示"暂无预约"，开发者 B 可能直接渲染一个空表格。

### B3. 验收标准中缺少多学生并发预约的覆盖（检查重点 3）

AC-08 覆盖了"两个学生约同一座位同一时段"的场景，但缺少：

- 学生 A 预约了 09:00-12:00，学生 B 在同一座位约 10:00-11:00——这应该冲突，但没有明确写
- 学生 A 预约了 09:00-12:00，学生 B 在同一座位约 12:00-14:00——这应该不冲突（衔接），但没有明确写
- 这些边界情况在状态流转第 12.3 节用示例描述了，但验收标准中没有对应的可操作用例

### B4. 索引描述可能造成误解（检查重点 6）

第 11.3 节描述索引：

> (SeatId, ReservationDate, StartTime, EndTime) 唯一组合索引，防止同一座位同一时段重复预约。

这个描述不完全准确。唯一索引只能阻止**完全相同的** (座位, 日期, 开始, 结束) 组合，**不能阻止时段重叠**。例如：

- 已存在：座位 A-01，2026-07-08，09:00-12:00
- 新插入：座位 A-01，2026-07-08，10:00-11:00（不同的 StartTime 和 EndTime，不会触发唯一索引冲突）

唯一索引可以作为辅助拦截，但重叠检测**必须靠应用层代码**（F4 中的冲突检测）。如果开发者看到这段描述以为"数据库会帮我挡住"，就会漏写应用层的冲突检测逻辑。

### B5. 页面清单有 1 个小口径偏差（检查重点 4）

MVP 确认文档第 5.1 节说"8 个页面 + 1 个共享布局"，PRD 第 7 节说"9 个视图文件（含 Layout + Error）"。多了一个 Error 页。

这在可接受范围内，但需要注意：**MVP 确认时没说 Error 页**。如果有严格的"口径对齐"要求，Error 页可以不在页面清单中单独列出（作为共享文件默认存在即可）。

## C. 必改项

### C1. 补充"已完成"状态，定义预约过期后的座位释放机制

**方案建议**（选择其中一个修改到 PRD）：

**方案一（推荐，改动最小）：** 在座位状态计算中加入"当天"过滤。

修改第 12.1 节和第 12.2 节：

```
座位状态计算规则：
- 空闲：当前没有与该座位关联的"预约中"记录，且记录的 ReservationDate 等于当天日期
- 已预约：存在与该座位关联的"预约中"记录，且 ReservationDate 为当天，且当前时间在 StartTime 到 EndTime 之间

说明：非当天的预约记录不影响座位状态的实时显示。
这样非当天的"预约中"记录不做任何处理，也不影响座位状态。
```

**方案二（更规范）：** 新增"已完成"状态，预约结束后自动变为该状态。这需要后端做判断，开发量稍大但更规范。

建议采用**方案一**，改动量最小。在 PRD 中添加以下说明：

```
> 注意：座位状态查询时必须过滤 ReservationDate = 当天日期。
> 非当天的预约记录（无论是"预约中"还是"已取消"）都不影响当天座位状态的判断。
```

并将此规则的验证加入验收标准：

```
AC-21（新增）：创建一个昨天的预约记录，今天查看座位列表 → 该座位状态应为"空闲"
```
